# Management API Vision

> Covers auto-generated management APIs and `ctl` CLIs derived from the Supervisor trait.
> Depends on the type system described in `health-vision.md`, `lifecycle-status-vision.md`,
> and `logging-vision.md`. The proc macro feasibility question is tracked in GitHub
> issue #14.

---

## Core insight: the types are already the API spec

Every operation a Supervisor exposes is already typed:
- `health_snapshot()` — returns `HashMap<KeyPath, ItemStatus>`
- `apply_desired(items)` — takes `impl IntoIterator<Item = T>`
- `query(source, since, until, level)` — returns `impl Iterator<Item = LogEntry>`

No separate API definition file. No hand-written endpoint code. The type system IS the
spec. A derive macro reads the Supervisor impl and generates the server and client
mechanically.

---

## The macro

```rust
#[derive(ManagementApi)]
#[management_api(
    socket = "/run/myapp/control.sock",
    name   = "myapp",
)]
struct MyOrchestrator { ... }
```

One annotation generates:
1. A Unix socket API server embedded in the Supervisor's run loop
2. A standalone `myapp-ctl` binary (or library) as the typed CLI client

## Generated server endpoints

| Method | Path | Maps to |
|---|---|---|
| `GET` | `/health` | `health_snapshot()` |
| `GET` | `/items` | iterate all items → `KeyPath + ItemStatus` |
| `GET` | `/items/{key_path}` | single item status by encoded `KeyPath` |
| `POST` | `/desired` | `apply_desired(items)` with JSON body |
| `POST` | `/reconcile` | trigger reconcile with current desired state |
| `GET` | `/logs` | `LogStore::query(source, since, until, level)` |
| `GET` | `/logs/{key_path}` | logs filtered to one item's subtree |
| `GET` | `/openapi.json` | auto-generated OpenAPI spec |

All request and response bodies are JSON. The `KeyPath` in URL segments uses the
base64url-encoded form (see `logging-vision.md`).

## Generated `ctl` CLI

```
myapp-ctl health
myapp-ctl health --watch          # poll and redisplay on change

myapp-ctl items
myapp-ctl items get <key-path>

myapp-ctl apply --file desired.json
myapp-ctl apply --stdin

myapp-ctl reconcile               # trigger one cycle

myapp-ctl logs
myapp-ctl logs --source <key-path>
myapp-ctl logs --since 1h --level warn
myapp-ctl logs --follow           # tail mode

myapp-ctl openapi                 # dump the OpenAPI spec
```

The CLI is generated by the same macro pass that generates the server. Every endpoint
produces a corresponding subcommand. Arguments are derived from the query/body types.
Output is JSON by default; `--format table` and `--format yaml` are generated where
meaningful.

This is the `kubectl` / `systemctl` pattern applied to any Supervisor. The `ctl` binary
knows the socket path from the annotation and connects automatically.

---

## Transport: Unix socket only, proxy for everything else

The framework generates a Unix socket server only. This is deliberate:

- The OS handles access control — filesystem permissions on the socket file replace auth
- No TLS, no token management, no session state — zero complexity inside the framework
- Works for local processes, containers (volume-mount the socket), and SSH tunnels

Remote access, authentication, and policy management are handled by a separate proxy layer
(nginx, caddy, or any HTTP reverse proxy) that the framework never touches. This avoids
turning the library into Django-level complexity while still supporting every deployment
topology.

```
Remote client
    → nginx / caddy (TLS + auth + policy)
        → Unix socket
            → ManagementApi server
                → Supervisor
```

---

## Ad-hoc vs long-running: same API, different mode

The socket is created when the Supervisor starts and removed when it stops. The mode is
irrelevant to the API shape:

- **Long-running server**: socket stays open indefinitely; `ctl` connects at any time
- **Ad-hoc / woken**: Supervisor starts (via D-Bus signal, cron, inotify, systemd socket
  activation), the socket exists for the duration, closes when done — `ctl` connects
  during the run window
- **One-shot**: Supervisor reconciles once and exits — `ctl reconcile` blocks until done,
  then the socket closes

The `ctl` binary works identically in all modes. The caller doesn't need to know which
mode the Supervisor is running in.

---

## Wakeup mechanisms

Because the server is Unix socket based, standard platform wakeup mechanisms work without
any framework support:

| Mechanism | How |
|---|---|
| **systemd socket activation** | systemd holds the socket fd; passes it to the process on first connection |
| **D-Bus signal** | a D-Bus service launches the Supervisor process on signal receipt |
| **cron / timer** | process starts, reconciles, exits |
| **inotify / file watch** | a small watcher process starts the Supervisor when a file changes |
| **manual `ctl` call** | `myapp-ctl reconcile` starts the process if socket-activated |

## What the proc macro needs to see

The macro operates on the Supervisor struct and its trait impl. It needs:
- The `type Item: Lifecycle` — to generate `apply_desired` request body schema
- `type HealthStatus` (= `ItemStatus`) — to generate `health_snapshot` response schema
- Whether the Supervisor implements `QueryableLogStore` — to generate log endpoints
- The socket path and name from the annotation attributes

The macro does not need to understand reconciler semantics — it only generates the network
glue between the typed method calls and the HTTP-over-Unix-socket transport.

## Prior art

| System | What it demonstrates |
|---|---|
| **dropshot** (Oxide) | REST API + OpenAPI generated from Rust types; production-grade |
| **tarpc** | RPC client + server generated from a Rust trait definition |
| **tonic** | gRPC client + server generated from proto; same derive-based pattern |
| **aide** + **axum** | OpenAPI spec generated from axum route types |
| **kubectl** / **systemctl** | The `ctl` pattern: a typed CLI client for a structured API |

`dropshot` is the closest in spirit — it generates REST + OpenAPI from Rust types, targets
Unix socket deployment, and is production-hardened. Whether to build on `dropshot` or
generate raw `axum`/`hyper` code is an open question (see GitHub issue #14).

## Open questions

- **Proc macro vs attribute macro vs build.rs codegen.** Proc macros see the AST but not
  semantic type information. Generating schema for associated types (`type Item`,
  `type HealthStatus`) may require attribute-level hints or a separate registration step.
  See GitHub issue #14.

- **`ctl` as a generated binary or a library.** Generating a full binary per Supervisor
  requires build system integration. A generated library + a thin main wrapper may be more
  practical.

- **Streaming endpoints.** `logs --follow` and `health --watch` require server-sent events
  or WebSocket. Does the macro generate these, or are they manual extensions?

- **Multi-Supervisor APIs.** If a process hosts multiple Supervisors, does each get its own
  socket, or do they share one with path-based routing? The `KeyPath` system already
  supports this naturally.

- **Schema evolution.** When `ItemStatus` or `type Item` changes, the generated API
  changes. How does the `ctl` binary handle version mismatches gracefully?
